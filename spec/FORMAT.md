# MSH1 바이너리 포맷 명세서

## 개요

MSH1은 고정 청크 dedup + 엔트로피 압축 기반 바이너리 컨테이너 포맷이다.
프레임 단위 스트리밍을 지원하며, 사전(dictionary)은 프레임을 걸쳐 누적된다.

## 매직 넘버

```
4바이트: 0x4D 0x53 0x48 0x31 ("MSH1")
```

## 프레임 구조

```
[프레임 헤더: 32바이트] [페이로드 크기: 4바이트] [압축 페이로드: N바이트] [CRC32: 4바이트 (선택)]
```

### 프레임 헤더 (32바이트, Little-Endian)

| 오프셋 | 크기 | 타입 | 필드 | 설명 |
|--------|------|------|------|------|
| 0 | 4 | bytes | magic | `MSH1` (0x4D534831) |
| 4 | 2 | uint16 | version | 포맷 버전 (현재: 1) |
| 6 | 2 | uint16 | flags | 플래그 비트 |
| 8 | 4 | uint32 | chunkSize | 청크 크기 (바이트) |
| 12 | 1 | uint8 | codecId | 코덱 ID |
| 13 | 3 | - | padding | 예약 (0x000000) |
| 16 | 4 | uint32 | origBytesLo | 원본 바이트 수 (하위 32비트) |
| 20 | 4 | uint32 | origBytesHi | 원본 바이트 수 (상위 32비트) |
| 24 | 4 | uint32 | dictEntries | 이 프레임의 새 사전 항목 수 |
| 28 | 4 | uint32 | seqCount | 시퀀스(인덱스) 수 |

### 페이로드 크기 (4바이트)

| 오프셋 | 크기 | 타입 | 필드 |
|--------|------|------|------|
| 32 | 4 | uint32 | payloadSize |

### 페이로드 (압축 해제 후)

```
[사전 섹션: dictEntries × chunkSize 바이트] [시퀀스 섹션: seqCount개 uvarint]
```

- **사전 섹션**: 이 프레임에서 새로 등장한 고유 청크들 (각 chunkSize 바이트)
- **시퀀스 섹션**: 원본 데이터의 청크 순서를 사전 인덱스(uvarint/LEB128)로 표현

### CRC32 (선택, 4바이트)

`flags & 0x0001`이 설정된 경우 프레임 끝에 CRC32 체크섬 추가.
계산 범위: 헤더(32B) + payloadSize(4B) + 압축 페이로드(NB)

## 플래그

| 비트 | 값 | 설명 |
|------|-----|------|
| 0 | 0x0001 | CRC32 체크섬 포함 |
| 1 | 0x0002 | 계층적 Dedup (HIERDEDUP) — 사전을 서브청크로 재-dedup |
| 2-15 | - | 예약 |

알 수 없는 플래그 비트가 설정된 파일을 읽으면 에러를 발생시켜야 한다.
이를 통해 구 버전 reader가 새 포맷 파일을 만났을 때 명확한 안내를 제공한다.

## 코덱 ID

| ID | 이름 | 설명 |
|----|------|------|
| 0 | none | 압축 없음 |
| 1 | gzip | gzip (level=1) |

## uvarint (LEB128)

시퀀스 섹션의 사전 인덱스는 unsigned varint로 인코딩된다.

```
값 0~127      → 1바이트: [0xxxxxxx]
값 128~16383  → 2바이트: [1xxxxxxx] [0xxxxxxx]
...
최대 5바이트 (uint32 범위)
```

## 압축 알고리즘

1. 입력을 `chunkSize` 바이트 고정 청크로 분할
2. 마지막 청크가 `chunkSize`보다 작으면 0x00으로 패딩
3. 각 청크를 SHA-256 해시하여 사전에서 중복 확인
   - 새 청크 → 사전에 추가 (글로벌 인덱스 할당)
   - 기존 청크 → 기존 인덱스 참조
4. `frameLimit` 바이트마다 프레임 경계 생성
5. 각 프레임의 페이로드(사전+시퀀스)에 코덱 압축 적용

## 복원 알고리즘

1. 프레임 헤더 파싱
2. 압축 페이로드 해제
3. 사전 섹션에서 새 청크를 전역 사전에 추가
4. 시퀀스 인덱스로 청크 순서 복원
5. `origBytes`로 마지막 청크 트리밍 (0 패딩 제거)

## 자동 청크 크기 감지 (auto 모드)

chunkSize를 `auto`로 지정하면 packer가 입력 데이터를 샘플링하여
최적 chunkSize를 자동으로 결정한다. 기본값이 `auto`이다.

### 감지 알고리즘

1. 샘플 추출: `min(전체 크기, 1MB)`
2. 후보 크기: `[32, 64, 128, 256, 512, 1024, 2048, 4096]` 바이트
3. 각 후보에 대해:
   - 샘플을 해당 크기로 분할 → SHA-256 해시 → 고유 청크 수 집계
   - 총비용 = `(고유청크수 × 청크크기)` + `(전체청크수 × varint바이트수)`
4. 총비용이 최소인 크기를 선택

### 스트리밍 모드

- 64KB 이상 데이터가 버퍼에 쌓이면 감지 수행
- 감지 전까지 프레임 방출 보류
- flush 시점에 미감지면 보유 데이터로 강제 감지

### 호환성

auto 모드는 Packer 측에만 존재한다.
출력 파일의 프레임 헤더에는 실제 선택된 chunkSize 값이 기록되므로,
Unpacker는 변경 없이 투명하게 처리한다.

## 상수 기본값

| 상수 | 값 |
|------|-----|
| DEFAULT_CHUNK_SIZE | auto (자동 감지, 폴백: 128 바이트) |
| DEFAULT_FRAME_LIMIT | 64 MB (67,108,864 바이트) |
| DEFAULT_CODEC | gzip |
| MIN_CHUNK_SIZE | 8 바이트 |
| MAX_CHUNK_SIZE | 16 MB (16,777,216 바이트) |
| AUTO_DETECT_CANDIDATES | [32, 64, 128, 256, 512, 1024, 2048] |
| AUTO_DETECT_SAMPLE_LIMIT | 1 MB |
| AUTO_DETECT_STREAM_MIN | 64 KB |
