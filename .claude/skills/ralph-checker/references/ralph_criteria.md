# Ralph Loop 적합성 기준

## 개요

Ralph Loop는 모든 작업에 적합하지 않습니다. 
다음 기준을 만족하는 작업에만 적용해야 합니다.

## 적합 조건 (모두 만족해야 함)

### 1. 명확한 완료 조건
- 테스트 통과/실패로 판단 가능
- 빌드 성공/실패로 판단 가능
- 특정 출력 존재 여부로 판단 가능
- UI 요소 존재/동작으로 판단 가능

### 2. 반복 가능한 검증
- 코드 실행으로 검증 가능
- 자동화된 테스트로 검증 가능
- 린트/타입체크로 검증 가능
- 스크린샷/DOM 검사로 검증 가능

### 3. 단계별 분할 가능
- 독립적으로 검증 가능한 단위로 분리
- 각 단계가 다음 단계에 의존하지 않음 (선형 의존은 가능)

## 부적합 케이스

###  주관적 판단 필요
- "보기 좋은 디자인으로 만들어줘"
- "사용하기 편한 UI로 개선해줘"
- "깔끔하게 정리해줘"

###  외부 API 응답 대기
- 결제 API 연동 (실제 결제 필요)
- 이메일 발송 확인 (실제 수신 필요)
- 외부 서비스 콜백 대기

###  매 단계 사용자 입력 필요
- "먼저 A를 만들고 보여줘, 그 다음 B를 어떻게 할지 알려줄게"
- 단계별 승인이 필요한 작업

###  모호한 완료 기준
- "잘 동작하면 완료"
- "에러 없으면 됨" (에러 범위 불명확)
- "적당히 만들어줘"

## 적합성 판단 예시

###  적합한 경우

```
오더: 로그인 페이지 만들어줘. 이메일/비밀번호 입력 후 /dashboard로 이동해야 해.

탈출조건:
1. npm run build 성공
2. /login 페이지 로드
3. 이메일 입력 필드 존재
4. 비밀번호 입력 필드 존재
5. 로그인 버튼 존재
6. 로그인 후 /dashboard로 리다이렉트
```

###  부적합한 경우

```
오더: 멋진 로그인 페이지 만들어줘.

문제: "멋진"의 기준이 주관적
해결: 구체적인 요구사항 추가 필요
```

## 적합성 변환 예시

부적합한 오더를 적합하게 변환하는 방법:

### Before (부적합)
```
"회원가입 폼 만들어줘"
```

### After (적합)
```
"회원가입 폼 만들어줘.
- 이름, 이메일, 비밀번호, 비밀번호 확인 필드
- 비밀번호 8자 이상 검증
- 이메일 형식 검증
- 모든 필드 필수
- 제출 시 /welcome으로 이동"
```
